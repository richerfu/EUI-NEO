### macOS Vulkan + SDL2 异常内存原因

定位过程：

- `vmmap -summary` 显示修复前 `vulkan-sdl2-release/gallery` 的额外内存主要在 `IOAccelerator (graphics)`：约 127.4 MiB；`MALLOC` 只有约 20 MiB 级别。
- `vulkan-glfw-release/gallery` 同场景 `IOAccelerator (graphics)` 约 3.4 MiB，说明不是 gallery 的普通 CPU 数据导致。
- 最小 SDL2 Vulkan surface/swapchain probe 约 76 MiB，`primitive_parity` 的 SDL2/Vulkan 约 100 MiB，说明不是 SDL2 + Vulkan 固定开销。
- 不同 gallery 页面触发的 `IOAccelerator` 占用不同，说明触发点在 EUI 的 Vulkan 渲染资源路径。

实际原因：

Vulkan 后端上传 text atlas / image texture 时创建 staging buffer，把它们放进 `pendingUploadBuffers_` / `pendingUploadMemories_`，原本只在下一次 `beginFrame()` 开头调用 `releasePendingUploads()` 释放。`gallery` 首帧会上传大量文本 atlas；静态 UI 首帧渲染完成后没有新的 dirty render，就不会再进入下一次 `beginFrame()`，导致 staging buffers 一直挂在进程里。macOS/MoltenVK 下这部分 host-visible 图形内存体现在 `IOAccelerator (graphics)`，所以 RSS 看起来大约多出 120 MiB。

修复：

`VulkanRenderBackend::makeCurrent()` 在没有 active frame 时检查上一帧 fence；如果 GPU 已完成，就释放 pending texture deletes 和 pending uploads。这样不早于 GPU 使用完成，也不依赖下一次真实渲染帧。

修复后复测，启动后等待 10 秒：

| Preset | 目标 | 平均 RSS | `IOAccelerator (graphics)` | `IOSurface` | 结论 |
| --- | --- | --- | --- | --- | --- |
| `vulkan-sdl2-release` | `gallery` | 112.7 MiB | 3.4 MiB | 16.0 MiB | 回到与 GLFW/Vulkan、OpenGL 组合接近的范围 |

因此，236 MiB 的实际原因是 Vulkan staging upload 生命周期错误，不是 SDL2 源码/库体积，也不是窗口后端天然比 GLFW 多占 120 MiB。

根因：

- 约 79-86 MiB RSS 在 `window_only_probe` 阶段已经存在，说明 RSS 大头首先来自 macOS 进程映射、Cocoa/GLFW/SDL2、动态库和系统共享页，而不是 EUI Runtime。
- SDL2/Vulkan/GLFW 的 `window_only_probe` physical footprint 只有约 15-16 MiB；RSS 明显高于 footprint，说明 RSS 会把大量共享库常驻页算进去。
- GLFW + OpenGL 的 `window_only_probe` physical footprint 已经达到 47.4 MiB，主要是窗口创建时 OpenGL-capable Cocoa surface / IOSurface 相关资源提前出现。
- 从 `window_only_probe` 到 `render_clear_probe`，OpenGL 主要增加 IOSurface / owned unmapped graphics；Vulkan 主要增加 swapchain/drawable 相关 footprint。这个阶段是渲染后端底噪，不是 DSL 或组件导致。
- 从 `render_clear_probe` 到 `empty_window`，RSS 变化很小，说明 AppRunner + DSL Runtime 本身不是 100 MiB 级别根因；但 Vulkan 路径的 physical footprint 会因 EUI render cache / backend resources 增加。
- EUI 自身普通 heap 在空窗口下约 5.8-6.7 MiB allocated；`eui_demo` 增加到约 11.8-14.6 MiB，gallery 增加到约 17.4-19.9 MiB。
- `eui_demo` 相比空窗口多出的约 14-19 MiB RSS，主要来自文本栈、默认字体/图标字体、text atlas、panel/button 绘制资源和窗口图标/CG image。
- gallery 相比 `eui_demo` 再多出的约 6-14 MiB RSS，主要来自更多文本 glyph、图片/图标资源、组件状态和更大的 surface/cache 压力。
- 用 RSS 评价轻量程度会高估框架成本；`Physical footprint` 和 `MALLOC allocated` 更能说明 EUI 自身成本。当前 macOS 最低 `empty_window` footprint 是 `vulkan-sdl2-release` 的 37.4 MiB，普通 heap allocated 约 5.8 MiB。

减少底噪的真实方向：

- RSS 底噪主要由系统和图形栈决定，不能通过组件层优化大幅消除；比较轻量程度时应同时看 Physical footprint 和 MALLOC allocated。
- macOS 上当前最低 footprint 组合是 SDL2 + Vulkan；如果目标是降低 macOS 系统内存压力，这个组合优于 OpenGL/GLFW。
- OpenGL/GLFW 的窗口创建阶段已经带来较高 IOSurface footprint；需要继续压底噪时，应优先考虑 SDL2 默认窗口后端或未来 Metal 后端，而不是继续抠 DSL 组件。
- EUI 可控部分主要是 render cache、窗口图标、text atlas、图片 CPU 缓存和 Vulkan 资源池；这些能降低内容增量，但不是 80 MiB RSS 底噪的主因。
- 真要进一步降低 macOS 图形底噪，长期方向是原生 Metal 后端；Vulkan 在 macOS 走 MoltenVK，OpenGL 走旧图形栈，都不是最低底噪路径。

### Windows Vulkan 交互内存波动原因

Windows 上交互时观察到 Vulkan 后端 Working Set 在约 90-230 MiB 间波动，OpenGL 后端没有同等级波动。该现象的主因仍是 staging upload，但触发方式不同于上面的静态首帧滞留：

- 文本 atlas 是共享页，灰度页尺寸为 2048x2048，单次完整上传约 4 MiB。
- 交互切页、hover 或打开新区域时，会让当前帧第一次绘制到一批新文本；每个文本图元在 `render()` 时可能追加新 glyph，使 atlas generation 递增。
- 修复前 Vulkan 在同一帧里按图元顺序发现 generation 变化，就会多次上传同一张逐步扩充的 atlas；这些 staging buffers 要等本帧提交并在后续 fence 完成后才能释放。
- 如果一帧内有几十次 atlas generation 递增，短时额外 Working Set 就会达到百 MiB 级别；这与 90-230 MiB 的波动量级一致。
- OpenGL 路径使用 `glTexImage2D` / `glTexSubImage2D` 更新 atlas，不会在进程 Working Set 中留下同样可见的一批 per-frame Vulkan staging allocations，所以同样交互下看起来更平稳。

修复：

Runtime 在实际提交绘制前增加文本 prepare pass：先遍历本次 dirty/full redraw 会绘制到的文本图元，提前完成 layout、glyph rasterize 和 atlas 写入；随后真正绘制时 Vulkan 只看到该帧最终 atlas generation，因此同一帧最多上传最终版本，而不是每追加一批 glyph 就重复上传整张 atlas。

## 内存由什么组成

进程 RSS 大致包含：

- 程序代码、静态库代码页、动态链接器和系统 framework 映射。
- C++ heap、线程栈、DSL Runtime 状态、layout/事件/动画缓存。
- FreeType、HarfBuzz、libpng、zlib、curl 等第三方库的运行状态。
- 字体 face、glyph atlas CPU 数据、文本 layout/metrics 缓存。
- 图片 CPU 解码缓存、GIF 帧缓存、remote/Bing 图片下载结果。
- 窗口后端本身的状态：GLFW 或 SDL2 的窗口、事件、IME、剪贴板、光标和平台集成资源。
- 渲染后端的 CPU 侧资源：shader/pipeline 管理、descriptor/texture 记录、vertex 数据、render cache 元数据。

OpenGL 额外包含：

- OpenGL context、driver 侧对象管理。
- framebuffer/cache、FBO、texture、VBO、shader program、默认 framebuffer/back buffer 相关资源。

Vulkan 额外包含：

- Vulkan loader、instance、physical/logical device、surface、swapchain、image view、framebuffer。
- command pool/buffer、semaphore、fence、render pass、pipeline、descriptor pool/set/layout。
- render cache image、backdrop/blur 中间图、text atlas image、image texture、staging buffer。
- driver 为 Vulkan 对象保留的 host memory。macOS 上还可能包含 MoltenVK/Metal 桥接层相关开销。

注意：

- `ps` 的 RSS 只反映进程常驻内存，不完整统计独立 GPU/driver 显存。
- macOS、Windows、Linux 的窗口系统、字体栈、图形驱动和 Vulkan loader 行为不同，数值不能跨机器直接比较。
- `EUI_RENDER_BACKEND=auto` 会按本机能力选择后端；做对比时必须使用明确的 `opengl-*` / `vulkan-*` preset。
- `gallery` 默认关闭 Glass/blur；打开 blur、Bing 图片或更多子窗口后，render cache、图片纹理和窗口资源都会继续增加。
